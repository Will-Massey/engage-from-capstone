# Performance & Operations Audit — 2026-07-07

_Note: file refs to `index.ts:1100+` predate the same-day bootstrap split —
that code now lives in `backend/src/app/*.ts` (semantics unchanged)._

## DB (P1 unless noted)

Missing composite indexes (all current indexes are single-column; hot queries
filter on 2-3): `Proposal(tenantId,status)`, `(tenantId,createdAt)`,
`(tenantId,sentAt)` (sentAt has no index at all), `(tenantId,status,renewalDate)`,
`(tenantId,validUntil)`, `Touchpoint(status,scheduledFor)`,
`ActivityLog(tenantId,createdAt)` + `(tenantId,entityType,entityId)`,
`EmailLog(tenantId,createdAt)`. Evidence: analytics funnel (8+ counts/req),
proposals list, chase/renewal/touchpoint jobs. One migration, effort S.
Search uses `contains/insensitive` (ILIKE %x%) — pg_trgm GIN later (M).

Unbounded queries: `jobs/emailAutomation.ts:320-342` (all SENT/VIEWED
proposals across ALL tenants w/ full includes), `renewalReminders.ts:214-233`
(per-row update loop → single UPDATE), `renewalProposalService.ts:129-141`
(JS-side window filter), `feeBenchmarkService.ts:102-124` (full scan per
request, uncached), `analytics.ts:829` (cap missing).

**Bugs found:** `routes/health.ts:11` creates a **second PrismaClient**
(doubles Neon connections, never disconnected — import shared).
`utils/jobLock.ts:30-42` holds a transaction (and its pooled connection) for
the whole job run, 10-min timeout — touchpoint engine can send 200 emails
inside it (switch to session advisory lock or shrink batches).
`jobs/touchpointEngine.ts:856-862` — `||`/ternary precedence bug: configured
`TOUCHPOINT_WEBHOOK_URL` is ignored; also double-fetches tenant.
`autoMigrateOnStartup` runs 5s AFTER listen — instance serves traffic against
possibly-unmigrated schema (prefer start-prod.mjs migrate-then-listen only).

## N+1

`emailAutomation` per-proposal ActivityLog query + 1s sleep per item (~8 min
inside job lock at 500 items) — prefetch with `in`. `renewalReminders`
per-tenant query loop. `touchpointEngine` 5-8 serial queries per touchpoint;
`proposalChaseJob` per-proposal/day dedupe query. Seeding per-row creates
(use createMany). Proposals list serializer is clean.

## Frontend bundle

Initial load ≈ **391 kB gzip JS**: index 217k, react 55k, data 42k,
framer-motion 42k, heroicons 9k, date-fns 7k (recharts 113k already lazy).
**Zero route-level `lazy()`** — App.tsx has 47 static page imports. Top splits:

1. public pages (`PublicProposalView`, `ClientPortal`, `AmlOnboarding`) — a
   client opening a signing link downloads the whole staff app; 2) builder
   cluster; 3) settings/subscription/legal; 4) analytics; 5) framer-motion off
   critical path.

## Caching

`utils/cache.ts` is dead code (imported once, only for disconnect; Redis-only,
no memory fallback). Cache-worthy: fee-benchmarks (24h), ai/status,
templates-meta counts, analytics dashboard (60s/tenant memo). Backend static
serving lacks `maxAge/immutable` on hashed assets. No compression middleware
(CF handles the public path).

## Observability/ops

- **Sentry dormant** (`SENTRY_DSN` not set in prod; render.yaml doesn't define
  it). Frontend has no Sentry; `ErrorBoundary.tsx:34` says "you could log to
  Sentry here". Sourcemaps disabled with nothing consuming them.
- **SIGTERM drops in-flight requests**: no `server.close()` drain, no
  `prisma.$disconnect()` (now in `app/shutdown.ts`). Every deploy hard-kills
  active requests.
- **Job failure alerting is logs-only** — 4 interval jobs `.catch(logger.error)`,
  no captureException, no lastRun on `/api/status`.
- Winston file transports write to ephemeral disk in prod (no disk resource) —
  drop them; two parallel logger modules used inconsistently; ~46 stray
  console.\* calls.
- Health checks good (real SELECT 1). Job advisory locks make scale-out safe.

## Cloudflare worker

**Zero edge caching** — every request incl. immutable hashed assets proxies to
Render. Fix: Cache API on `isStaticAsset` + `Cache-Control: public,
max-age=31536000, immutable` on `/assets/*`; short TTL on index.html;
`observability: enabled` in wrangler.jsonc.
